# Federated Learning with MLOps

This project trains a vibration-based anomaly detection model using Federated Learning (Flower) and wraps the training workflow with MLOps tooling (Prefect + MLflow) and a web control API (FastAPI).

The frontend is a separate React/Vite app (already deployed on Vercel), while the backend orchestrates training runs and exposes endpoints that the UI calls.

## What This Project Does

- Trains a shared global model with 3 simulated clients (M01, M02, M03) using Flower FedAvg.
- Loads either pre-consolidated H5 datasets or raw per-machine H5 files.
- Logs per-round client/server metrics and artifacts with MLflow.
- Orchestrates distributed FL runs using a Prefect flow.
- Exposes a FastAPI backend to trigger pipeline runs from the frontend.

## Tech Stack

- Model and training: TensorFlow, NumPy, h5py
- Federated learning: Flower
- Orchestration: Prefect
- Experiment tracking: MLflow
- Backend API: FastAPI + Uvicorn
- Frontend: React + Vite (in frontend folder)
- Containerization: Docker + Docker Compose

## High-Level Architecture

1. FastAPI backend receives UI requests.
2. Backend triggers commands from manage_pipeline.py.
3. Prefect flow starts one Flower server and 3 Flower clients as subprocesses.
4. Clients train locally and return updates to server each round.
5. Metrics and artifacts are logged to MLflow and report files.
6. Backend serves current status and latest run summary to frontend.

## Repository Structure

- web_backend.py: FastAPI service used by the frontend.
- manage_pipeline.py: CLI entrypoint to setup, run, and inspect pipeline operations.
- flows/distributed_fl_pipeline.py: Prefect flow that coordinates distributed FL.
- server.py: Flower server with FedAvg strategy and global model aggregation.
- client.py: Flower client implementation with local training and MLflow logging.
- model1.py: CNN model definition for vibration data.
- consolidate_data.py: Data consolidation helper for H5 datasets.
- data_consolidated/: Consolidated machine and test datasets.
- reports/: Per-round and evaluation report artifacts.
- pipeline_logs/: Runtime logs generated by orchestrated runs.
- frontend/: React/Vite frontend app.
- Dockerfile, docker-compose.yml: Container and local multi-service setup.

## Data Expectations

The code supports two data modes:

1. Consolidated mode (preferred)
- data_consolidated/M01_consolidated.h5
- data_consolidated/M02_consolidated.h5
- data_consolidated/M03_consolidated.h5
- data_consolidated/test_consolidated.h5

2. Raw folder mode
- new_train/M01/{good,bad}/*.h5
- new_train/M02/{good,bad}/*.h5
- new_train/M03/{good,bad}/*.h5
- new_test/{good,bad}/*.h5

## Local Setup (Python)

1. Create and activate virtual environment.
2. Install dependencies.
3. Start backend API.

Example commands:

Windows PowerShell:
- python -m venv venv
- .\\venv\\Scripts\\Activate.ps1
- pip install -r requirements.txt
- uvicorn web_backend:app --host 0.0.0.0 --port 8000

## Pipeline CLI

manage_pipeline.py supports the following commands:

- setup: Validate environment, dependencies, and data directories.
- mlflow: Start local MLflow tracking server.
- run: Execute distributed FL Prefect pipeline.
- config: Print detected pipeline configuration.
- experiments: List MLflow experiments and sample runs.
- deploy: Trigger Prefect deployment by name.

Examples:

- python manage_pipeline.py setup
- python manage_pipeline.py mlflow
- python manage_pipeline.py run --experiment-name distributed_fl --num-rounds 3 --timeout-minutes 30
- python manage_pipeline.py experiments

## Backend API Endpoints

Base URL: http://localhost:8000

- GET /status
	- Returns status of pipeline, mlflow, and prefect subprocesses.
- POST /run-pipeline
	- Starts distributed FL pipeline run.
- POST /start-mlflow
	- Starts MLflow server via manage_pipeline.py.
- POST /start-prefect
	- Starts Prefect server.
- POST /stop/{name}
	- Stops a running service by name.
- GET /latest-result
	- Parses latest report and returns summary (accuracy, rounds, filename).

## Docker Usage

Build backend image:
- docker build -t fl-mlops-backend .

Run backend container:
- docker run -p 8000:8000 fl-mlops-backend

Run all services with compose:
- docker compose up --build

Compose services:
- backend on 8000
- mlflow on 5000
- prefect on 4200

## Deployment Notes

- Frontend: hosted on Vercel.
- Backend: intended for Azure App Service (Linux container) with ACR image.
- CORS in web_backend.py currently allows https://mlops-pipeline-fl.vercel.app.

If your Vercel domain changes, update allow_origins in web_backend.py and redeploy backend.

## Outputs and Artifacts

- Final global model weights: final_model_weights.weights.h5
- Best model snapshot: best_model/best_model.weights.h5
- MLflow artifacts and runs: mlruns/
- Human-readable text reports: reports/
- Runtime process logs: pipeline_logs/

## Notes

- The backend launches long-running processes from API endpoints. For production, use a robust process supervisor and persistent storage for artifacts.
- Some dependencies are heavy (TensorFlow, MLflow). Container builds and cold starts can take time on low-resource tiers.
